# repo-alive

Makes any codebase self-explanatory. Default mode is a TUI conversation in Claude Code. An experimental HTML canvas is also available.

## How it works

```
/repo-alive
    ↓
Check .repo-alive/fingerprint.json vs git HEAD
    ↓ stale or missing
Analyze repo  →  write .repo-alive/ manifests
    ↓ fresh
Enter TUI chat — answer questions from manifests + source files
```

Analysis runs once per git HEAD. Results persist in `.repo-alive/` and are reused until the repo changes.

## Modes

```
/repo-alive              — analyze if stale, then TUI chat
/repo-alive analyze      — force re-analysis, then TUI chat
/repo-alive --html       — [EXPERIMENTAL] analyze, then serve canvas at localhost:4311
```

**TUI mode** (default): no runtime deps. Claude reads manifests and source files, answers questions, cites `file:line`.

**HTML mode** (experimental): starts a local Node.js server. Requires `node >= 18` + `npm`. Claude installs deps automatically on first run.

## Files

```
skill.md      Analysis algorithm + chat instructions (Claude Code reads this)
server.js     Local HTTP + WebSocket server (~360 lines, dep: ws)
client.js     Browser-side graph renderer (~340 lines, no deps)
canvas.html   Generic HTML shell served by server.js
package.json  { "dependencies": { "ws": "^8.0.0" } }
```

`server.js` and `canvas.html` are only used in `--html` mode. TUI mode uses none of these.

## Output: `.repo-alive/`

Generated by the analysis phase. Required by both TUI and HTML modes.

```
fingerprint.json      { git_head, generated_at, tool }  — freshness check
graph.json            { nodes: [...], edges: [...] }
ownership.json        { "<file>": "<node-id>" }
reverse-index.json    { "<node-id>": { owned_files, scenario_refs } }
nodes/
  <node-id>.json      Full node manifest (summary, files, interfaces, connections)
scenarios/
  <scenario-id>.json  Three-view scenario (structure / behavior / data)
```

## Node manifest schema

```json
{
  "id": "l0:api",
  "name": "API Server",
  "level": "L0",
  "root_path": "services/api",
  "summary": "...",
  "owned_files": ["services/api/main.go", "..."],
  "key_files": [{ "path": "...", "reason": "..." }],
  "interfaces": { "receives": [...], "sends": [...] },
  "connections": [{ "to": "l0:db", "type": "storage", "confidence": 0.9 }],
  "confidence": 0.87
}
```

## Failure modes

| Situation | Behavior |
|-----------|----------|
| No git repo | Uses `pwd` as root, fingerprint uses `NO_GIT` |
| No `.repo-alive/` | Analysis runs automatically |
| Port 4311 in use (HTML mode) | Existing process is killed before starting |
| Node not installed (HTML mode) | Error on `npm install` — install Node >= 18 |
| >20% of nodes affected by file changes | Full re-analysis triggered |

## Zero framework assumptions

The analysis uses only generic signals — file structure, entrypoints, import/export patterns, root manifests (`package.json`, `go.mod`, `Cargo.toml`, etc.). Works on any language and any project layout.
